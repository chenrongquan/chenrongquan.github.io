gRPC | 简单认识

2021年6月15日

引用

gRPC 和 protocol buffers
gRPC可以使用protocol buffers作为接口定义语言(IDL)和底层信息交换格式。

什么是gRPC

在gRPC中 客户端应用程序可以像调用本地对象的方法一样，调用不同机器上的服务端应用程序上的方法，使得创建分布式应用和服务更容易。
gRPC基于服务定义的思想，指定能够被远程调用的方法(包含参数和返回类型)，服务端实现该接口，并运行一个gRPC服务端来处理客户端调用。在客户端，有一个存根，提供与服务端相同的方法。

gRPC客户端和服务端可以在多种环境中运行和相互通信，并且可以用任何gRPC支持的语言来编写。

使用protocol buffers

gRPC默认使用protocol buffers，是google开源的一套成熟的结构数据序列化机制。
使用protocol buffers步骤

  1. 要在proto文件中定义好要序列化数据的结构，是一个带有.proto扩展名的文本文件。
  2. protocol buffers中的数据被构造为 消息，每个消息都是一个逻辑对象，包括一系列key-value对
  3. 使用protoc编译器从定义中生成数据访问类，为每个字段提供了访问方法 如 name() 或 set_name()；以及整个结构的序列化和反序列化方法。

protocol buffers版本

proto3；
虽然可以使用 proto2（当前默认的协议缓冲区版本），但还是建议将 proto3 与 gRPC 一起使用，因为它可以支持所有 gRPC 支持的语言，避免proto2客户端和proto3服务端通信产生的兼容性问题。

核心概念

服务定义

RPC基于服务定义的思想，指定远程调用的方法的参数和返回类型。
默认情况下，使用protocol buffer作为IDL，用于描述服务接口和载荷消息。

服务定义：在helloworld.proto里用 protocol buffers IDL定义的
Greeter服务有个远程方法SayHello，可以让服务端从远程客户端接收HelloRequest消息后，返回HelloReply

gRPC定义了四种服务方法

  • 一元RPC
    客户端发送单个请求到服务端，并返回单个响应，就像一个函数调用
  • 服务端流式RPC
    客户端发送单个请求到服务端，服务端返回流式的一系列消息。客户端持续读取返回流直至没有更多的消息。gRPC保证消息顺序
  • 客户端流水RPC
    客户端持续向服务端发送一些列流式消息，发送完成后。客户端等待服务端处理并返回。gRPC保证消息的顺序性。
  • 双向流式RPC
    使用读写流发送一系列消息，两个流独立运行。因此客户端和服务端可以按照任何顺序进行读写。例如 服务端可以在写入响应之前等待接收所有的客户端消息。或者也可以交替读取和写入消息。或者其他组合。保留每个流中消息的顺序。

    使用API

    gRPC用户通常在客户端调用这些API，并在服务器端实现相应的API。
  • 服务器端，服务器实现服务声明的方法并运行gRPC服务器来处理客户端调用，gRPC解码请求，执行服务方法，加密服务响应
  • 客户端，客户端有一个stub的本地对象，实现与服务相同方法。客户端可以在本地对象上调用这些方法。将调用参数包装在protocol buffer消息类型中。gRPC负责将请求发送到服务端，返回服务端响应。

异步 vs 同步

同步RPC调用阻塞直至接收到服务端响应，最接近于 RPC 所追求的过程调用的抽象。另一方面，网络本质上是异步的，在许多情况下，能够不阻塞当前线程的情况下启动RPC很有用。

RPC生命周期

一元RPC

  • 一旦客户单调用了Stub方法，服务器就会收到通知。RPC已经被调用，包括客户端的元数据、方法名称、指定的截止日期。
  • 服务端立即返回自己的初始元数据(必须在任何响应之前发送)，或者等待客户端的请求消息。
  • 一旦服务器接收到客户端的请求消息，就会执行创建响应，然后将响应连同态详细信息（状态代码和可选状态消息）和可选的尾随元数据一起返回给客户端。
  • 如果响应状态为OK，则客户端得到响应，在客户端完成调用。

    服务器流式RPC

  • 服务器返回消息流以响应客户端的请求。发送完所有消息后，服务器的状态详细信息（状态代码和可选状态消息）和可选的尾随元数据将发送到客户端。这样就完成了服务器端的处理。一旦客户端拥有服务器的所有消息，它就完成了。

    客户端流式RPC

    客户端向服务器发送消息流而不是单个消息。服务器用一条消息（连同它的状态详细信息和可选的尾随元数据）进行响应，通常不一定是在它收到所有客户端的消息之后。

    双向流式RPC

    在双向流式 RPC 中，调用由调用方法的客户端和接收客户端元数据、方法名称和截止日期的服务器发起。服务器可以选择发回其初始元数据或等待客户端开始流式传输消息。
    客户端和服务器端流处理是特定于应用程序的。由于两个流是独立的，客户端和服务器可以按任意顺序读写消息。例如，服务器可以等到收到所有客户端的消息后再写入消息，或者服务器和客户端可以玩“乒乓”——服务器收到请求，然后发回响应，然后客户端发送基于响应的另一个请求，依此类推。

截止日期/超时

gRPC 允许客户端指定在 RPC 因DEADLINE_EXCEEDED错误终止之前他们愿意等待 RPC 完成多长时间。在服务器端，服务器可以查询特定的 RPC 是否超时，或者还剩下多少时间来完成 RPC。
指定截止日期或超时是特定于语言的：一些语言 API 根据超时（持续时间）工作，而某些语言 API 根据截止日期（固定时间点）工作，可能有也可能没有默认截止日期。

RPC终止

在 gRPC 中，客户端和服务器都对调用的成功做出独立和本地的判断，它们的结论可能不一致。这意味着，例如，您可能有一个 RPC 在服务器端成功完成（“我已经发送了我所有的响应！”）但在客户端失败（“响应在我的截止日期之后到达！”）。服务器也有可能在客户端发送其所有请求之前决定完成。

取消RPC

客户端或服务器都可以随时取消 RPC。取消会立即终止 RPC，以便不再进行进一步的工作。

取消之前所做的更改不会回滚。

元数据

元数据是键值对列表形式的特定 RPC 调用的信息（例如身份验证详细信息），其中键是字符串，值通常是字符串，但也可以是二进制数据。元数据对 gRPC 本身是不透明的——它允许客户端提供与服务器调用相关的信息，反之亦然。对元数据的访问取决于语言。

频道

gRPC 通道提供到指定主机和端口上的 gRPC 服务器的连接。它在创建客户端存根时使用。客户端可以指定通道参数来修改 gRPC 的默认行为，例如打开或关闭消息压缩。通道具有状态，包括connected和idle。
gRPC 如何处理关闭通道取决于语言。某些语言还允许查询通道状态。
