apm | 上下文管理器ContextManager

2021年6月14日

Span

Span在分布式追踪系统中是一个重要而且得到公认的概念。关于Span可参考Google Dapper Paper和 OpenTracing

Skywalking从2017年开始支持OpenTracing和OpenTracing API; 在Skywalking中Span概念与谷歌的Dapper论文 和 OpenTracing相似，也去做了扩展。
有三种类型的Span

  1. Entry Span：代表服务提供者(Service Provider)，它也是服务端的一个端点。作为一个APM系统，我们的目标是应用服务。因此，几乎所有的服务和MQ消费者都是EntrySpan。
  2. Local Span：代表一个正常的Java方法内部调用，与远程服务无关。它就是一个MQ生产者/消费者，也不是服务(如 HTTP服务)的调用方/接收方
  3. Exit Span：代表服务的客户端或者MQ的生产者，早期的Skywaking称之为LeafSpan。举个例子，通过JDBC访问DB 和 读取 Redis/Memcached 是 典型的ExitSpan。

ContextCarrier

为了实现分布式跟踪，必须绑定跨进程跟踪，并且上下文必须跨进程传播。这是ContextCarrier产生的原因。
以下是有关如何在 A->B 分布式调用中使用 ContextCarrier 的步骤。

  1. 在客户端创建一个新的而且为空的ContextCarrier
  2. 调用 ContextManager#createExitSpan 创建一个ExitSpan 或者 使用 ContextManager#inject 初始化这个 ContextCarrier
  3. 将ContextCarrier所有元素绑定到Headers上(如 HTTP HEAD)、附加信息(如 Dubbo PPC框架)或者消息(例如 Kafka)
  4. ContextCarrier通过服务调用传递到服务端
  5. 服务端从heads中获取所有的元素、附加信息、消息
  6. 通过调用ContextManager#createEntrySpan 创建EntrySpan 或者 使用ContextManager#extract 来绑定客户端与服务端

Apache HTTPComponent 客户端插件 和 Tomcat7 服务端插件

在客户端使用Apache HTTPComponent插件

在服务端使用Tomcat7 服务端插件

ContextSnapshot

除了跨进程跟踪，还必须支持跨线程跟踪。例如，异步进程（内存 MQ）和批处理在 Java 中都很常见。跨进程和跨线程跟踪非常相似，因为它们都需要传播上下文, 但是跨线程跟踪不需要序列化。
以下是跨线程传播的三个步骤：

  • 使用ContextManager#capture获取 ContextSnapshot 对象
  • 让子线程通过方法参数或由现有参数携带来访问ContextSnapshot
  • 在子线程中使用 ContextManager#continued 方法

核心APIS

ContextManager

ContextManager的主要APIS

  1. Create EntrySpan
    public static AbstractSpan createEntrySpan(String endpointName, ContextCarrier carrier)
    根据operationName(比如 服务名、uri) 和 ContextCarrier 来创建EntrySpan

  2. Create LocalSpan
    public static AbstractSpan createLocalSpan(String endpointName)
    根据operationName(如 方法的全限定名)来创建LocalSpan

  3. Create ExitSpan
    public static AbstractSpan createExitSpan(String endpointName, ContextCarrier carrier, String remotePeer)
    根据operationName(比如 服务名、uri) 和 新的ContextCarrier 和 调用方地址(如 ip+port 或 hostname+port)来创建ExitSpan

AbstractSpan

除了设置操作名称、标签和日志之外，还必须设置两个属性，即组件和层。 这对于 EntrySpan 和 ExitSpan 尤其重要。
SpanLayer是Span的类型，有5个值

  1. UNKNOWN(default)
  2. DB
  3. RPC_FRAMEWORK(为 RPC 框架设计，而不是普通的 HTTP 调用)
  4. HTTP
  5. MQ

组件 ID 由 SkyWalking 项目定义和保留。对于组件名称/ID 的扩展，请遵循组件库定义和扩展文档(component library definitions and extensions)[https://github.com/apache/skywalking/blob/master/docs/en/guides/Component-library-settings.md]

Special Span Tags

所有标签在跟踪视图都是可用的。 同时，在 OAP 后端分析中，一些特殊的标签或标签组合提供了其他高级功能。

Tag key status_code
该值应该是一个整数。 OAL 实体的响应码对应这个值。

Tag keys db.statement 和 db.type
db.statement 的值代表数据库语句，如SQL，如果值为空则为[No statement]/+span#operationName。当ExitSpan包含此标签时，OAP 会根据 agent-analyzer/default/maxSlowSQLLength 对慢SQL进行采样。慢SQL的阈值由agent-analyzer/default/slowDBAccessThreshold定义

扩展逻辑端点：Tag key x-le
逻辑端点是一个概念，不代表真正的 RPC 调用，但需要统计信息。 x-le 的值应该是 JSON 格式。 有两种选择：

  1. 定义一个分离的逻辑端点。 提供其自己的端点名称、延迟和状态。 适用于EntrySpan 和 LocalSpan。
  2. 声明代表逻辑端点的当前本地跨度。

高级APIS

Aysnc Span APIS

Span 中有一组高级 API，专门为异步用例设计。 当需要在另一个线程中设置Span的标签、日志和属性（包括结束时间）时，您应该使用这些 API。

  1. 在原始上下文中调用 #prepareForAsync
  2. 当当前线程任务完成时，在原始上下文中运行 ContextManager#stopSpan
  3. 将Span传递给任意其他线程
  4. 完成上述步骤后，在任意线程中调用``#asyncFinish`。
  5. 当所有Span完成#prepareForAsync ，跟踪上下文将完成并告知后端(基于 API 执行次数)
